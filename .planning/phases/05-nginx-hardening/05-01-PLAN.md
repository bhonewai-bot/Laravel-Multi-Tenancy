<task type="auto">
  <name>Task 2: Add security headers to dev and prod nginx configs</name>
  <files>docker/nginx/conf.d/app.conf, docker/nginx/conf.d/prod/app.conf</files>
  <read_first>

  <action>
Add security headers to BOTH nginx config files. The critical nginx behavior: `add_header` in a location block overrides ALL server-level headers. Since the PHP location block will need its own `add_header` in Plan 05-02 (for Cache-Control), all security headers must be repeated in every location block that has any `add_header` directive.

**Strategy:** Add all security headers at the server level, AND in every location block that will have its own `add_header` in Plan 05-02. This ensures headers survive the inheritance override.

### Dev config (`docker/nginx/conf.d/app.conf`)

Add these lines after `root /var/www/public;` in the existing server block (per D-01, D-02, D-04, D-05, D-09, D-10):

```
    # Security headers
    add_header Content-Security-Policy-Report-Only "script-src 'self' 'unsafe-inline' 'unsafe-eval'; connect-src 'self' ws: wss:;" always;
    add_header Strict-Transport-Security "max-age=300; includeSubDomains" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
```

Note: Dev CSP includes `connect-src 'self' ws: wss:` for Vite HMR WebSocket in development (per D-02).

The existing PHP location block (`location ~ \.php$`) does NOT currently have any `add_header` directives, so server-level headers will apply to PHP responses. No changes needed to the PHP block for now -- Plan 05-02 will add Cache-Control headers there and must repeat these security headers at that time.

### Prod config (`docker/nginx/conf.d/prod/app.conf`)

In the HTTP (port 80) server block: Add after `root /var/www/public;` and before the location blocks:

```
    # Security headers
    add_header Content-Security-Policy-Report-Only "script-src 'self' 'unsafe-inline' 'unsafe-eval'; connect-src 'self';" always;
    add_header Strict-Transport-Security "max-age=300; includeSubDomains" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
```

Note: Prod CSP uses `connect-src 'self';` only (no ws:/wss:) per D-03.

In the HTTPS (port 443) server block: Add the same four headers after `root /var/www/public;` and before the SSL directives, but adjust the CSP to be the same strict policy:

```
    # Security headers
    add_header Content-Security-Policy-Report-Only "script-src 'self' 'unsafe-inline' 'unsafe-eval'; connect-src 'self';" always;
    add_header Strict-Transport-Security "max-age=300; includeSubDomains" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
```

Both prod server blocks get identical security headers. The `always` flag ensures headers are present even on error responses (4xx, 5xx).
