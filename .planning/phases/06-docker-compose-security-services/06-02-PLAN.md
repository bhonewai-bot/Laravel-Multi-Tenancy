  <action>Add a new `scheduler` service to docker-compose.prod.yml. The service definition must include:

1. **Same build as app** (per D-03): `build: { context: ., dockerfile: Dockerfile }` — reuses the production image
2. **Same env_file as app** (per D-04): `env_file: - .env`
3. **Same environment overrides as app**: `environment: APP_ENV: production, APP_DEBUG: "false"`
4. **Command** (per D-05): `command: php artisan schedule:work` — Laravel's built-in scheduler loop that checks every minute for due commands
5. **Restart policy** (per D-06): `restart: unless-stopped`
6. **Same working_dir as app**: `working_dir: /var/www`
7. **Same depends_on as app** (per D-04): depends on mysql with `condition: service_healthy`
8. **Same network**: `networks: - appnet`

Place the scheduler service between the queue and mysql services in the file (maintaining logical grouping: app services, then infrastructure services).

The final service block structure:

```yaml
  scheduler:
    build:
      context: .
      dockerfile: Dockerfile
    cap_drop:
      - ALL
    security_opt:
      - no-new-privileges
    env_file:
      - .env
    environment:
      APP_ENV: production
      APP_DEBUG: "false"
    restart: unless-stopped
    working_dir: /var/www
    command: php artisan schedule:work
    depends_on:
      mysql:
        condition: service_healthy
    deploy:
      resources:
        limits:
          memory: 256M
          cpus: "0.5"
    networks:
      - appnet
```

Note: This task also adds security constraints (cap_drop, security_opt per D-01/D-02) and resource limits (256M/0.5 CPU per D-09) to the new scheduler service so it is fully hardened from the start.
