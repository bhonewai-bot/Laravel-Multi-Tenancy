The tenancy foundation (Stancl Tenancy) is correctly configured with proper database isolation. The test suite exists and exercises real tenant flows. However, the application layer on top — the module system, authorization, and Cloudflare integration — was rushed and contains dangerous shortcuts. The Docker setup has structural problems (root execution, dead entrypoint, missing security headers).

## Critical Security Issues

These three issues must be fixed before any public deployment. Each one is an exploitable vulnerability.

### CRITICAL-1: No Central Admin Authorization

**Severity:** Critical — any authenticated user can provision tenants and upload executable code.

**File:** `routes/web.php:14-35`

Every central route is protected only by the `auth` middleware. There is no admin role check, no Gate, no policy:

```php
// routes/web.php:14
Route::middleware('auth')->group(function () {
    Route::resource('tenants', TenantController::class);       // anyone can create tenants
    Route::post('/modules', [ModuleController::class, 'store']); // anyone can upload ZIPs
    Route::post('/module-requests/{moduleRequest}/approve', ...); // anyone can approve modules
});
```

The `TenantStoreRequest::authorize()` check at line 14 is `return (bool) $this->user()` — meaning "is logged in," not "is admin." The `CentralAdminService` creates a super-admin on boot but nothing enforces that only the super-admin can access central routes.

Any user who registers or discovers the login page can create tenants, upload module ZIPs containing arbitrary PHP, and approve module installations across all tenants.

**Required fix:** Add a central admin Gate or middleware. Check that the authenticated user's email matches the configured `CENTRAL_SUPERADMIN_EMAIL`. Apply it to all central CRUD routes via a route group.

---

### CRITICAL-2: Module ZIP Upload is Remote Code Execution

**Severity:** Critical — full server compromise possible via file upload.

**Files:**

- `app/Services/ModuleZipInspector.php:42-45`
- `app/Http/Controllers/ModuleController.php:49-68`

The inspection step detects dangerous file extensions (PHP, phtml, sh, exe, bat, cgi) but intentionally does not block them:

```php
// ModuleZipInspector.php:43
// "allow php only inside a valid module package; do not block here"
```

After passing inspection unblocked, the ZIP is extracted directly into `base_path('Modules/')` — within the application root and web-accessible directory. Any PHP file inside the ZIP becomes executable code with full application privileges.

Combined with CRITICAL-1 (no admin authorization), any authenticated user can upload a ZIP containing a PHP webshell, then access it via the web server.

**Required fix:** At minimum, gate module upload behind admin-only access. Then add at least one of: code signing, file content allowlisting, extraction outside the web root, or a module registry with vetted packages.

---

### CRITICAL-3: Module State Race Condition (JSON Blob)

**Severity:** Critical — concurrent install/uninstall operations corrupt module state.

**File:** `app/Services/TenantModuleRegistry.php:185-214`

Installed modules and operation status are stored as JSON arrays in the tenant record's `data` column. Every mutation calls `refresh()` then `save()` in a non-transactional read-modify-write cycle:

```php
// TenantModuleRegistry.php:201-214
private function upsertModuleOperation($tenant, string $slug, ...): void
{
    $tenant->refresh();                    // read current JSON
    $operations = $this->getModuleOperations($tenant);
    $operations[$slug] = [...];            // modify in memory
    $tenant->setAttribute('module_operations', $operations);
    $tenant->save();                       // write back — last write wins
}
```

The docblock at line 195 acknowledges this: **"WARNING: This is last-write-wins state."** Acknowledging a race condition in a comment does not fix it.

If two queue workers process operations for the same tenant concurrently (e.g., install module A while module B is still installing), one operation's state gets silently clobbered. The cache lock in `TenantModuleInstaller` only serializes the same tenant+module pair, not cross-module operations for the same tenant.

**Required fix:** Move `installed_modules` and `module_operations` to a dedicated pivot/junction table with database-level transactional guarantees.

## Architecture Critique

### Major Issues

#### MAJOR-1: Duplicate Host Verification Systems

**Files:** `app/Support/HostResolver.php` and `app/Services/TenantDomainService.php`

Two completely separate implementations of host verification logic exist side by side:

| Method                     | HostResolver           | TenantDomainService        |
| -------------------------- | ---------------------- | -------------------------- |
| Is this a central host?    | `isCentralHost()`      | `isCentralDomain()`        |
| Find tenant domain         | `findTenantDomain()`   | `isVerifiedCustomDomain()` |
| Is it a primary subdomain? | `isPrimarySubDomain()` | `isPrimarySubDomain()`     |
| Can serve traffic?         | `canServeTenantHost()` | `canUseAsTenantDomain()`   |

Implementation diverges: `HostResolver::isPrimarySubDomain` uses `str_ends_with`, while `TenantDomainService::isPrimarySubDomain` uses string concatenation with `==` comparison. When one gets updated and the other doesn't, you get a security hole where one middleware trusts a domain the other rejects.

**Required fix:** Consolidate on `TenantDomainService` (it's more complete). Delete `HostResolver`.

---

#### MAJOR-2: EnsureModuleInstalled Identity-Map No-Op

**File:** `app/Http/Middleware/EnsureModuleInstalled.php:37-41`

```php
// The array is normalized before comparison so route parameters can stay case-insensitive.
$installedModules = array_map(
    fn ($item) => $item,     // <-- maps every element to ITSELF
    $installedModules
);
```

`fn ($item) => $item` is the identity function. It does nothing. The comment says it normalizes for case-insensitive comparison, but the lambda was never written. At line 45, the comparison uses `in_array($targetModule, $installedModules, true)` with `$targetModule = Str::lower($module)`. Since the array was never lowercased and the comparison is strict, this check fails for any installed module whose stored slug has uppercase characters.

**Required fix:** Change to `array_map(fn ($item) => strtolower($item), $installedModules)`. Better yet, normalize on write in `TenantModuleRegistry`.

---

#### MAJOR-3: CreateTenantAction Has No Database Transaction

**File:** `app/Actions/Tenants/CreateTenantAction.php:15-36`

```php
public function execute(array $data): Tenant
{
    $tenant = Tenant::create([...]);         // Step 1
    $domainModel = $tenant->domains()->create([...]);  // Step 2
    $this->syncCloudflareDomainAction->execute(...);   // Step 3
    return $tenant;
}
```

No database transaction wraps these operations. `UpdateTenantAction` correctly uses `DB::transaction()`, but the creation path does not. If domain creation fails at step 2 after the tenant was created at step 1, you get an orphaned tenant with no domain — inaccessible forever. Worse, Stancl's `TenantCreated` event fires on step 1, triggering `CreateDatabase`, `MigrateDatabase`, and `SeedDatabase` jobs. If step 2 fails, you have a provisioned database with no way to route to it.

**Required fix:** Wrap the entire sequence in `DB::transaction()`. Dispatch Cloudflare sync as a queued job, not inline.

---

#### MAJOR-4: Tenant Model is Empty

**File:** `app/Models/Tenant.php`

The model is 13 lines with zero custom attributes, scopes, accessors, or relationships. All tenant data (name, email, description) lives in the `data` JSON column via Stancl's magic attribute resolution. This means:

- No type casting on tenant fields
- `$tenant->name` works via magic but `$tenant->getAttribute('name')` may not
- Eloquent observers, accessors, and mutators don't fire for JSON-stored fields
- No query scopes (active tenants, tenants by plan, etc.)
- No relationship definitions to Module or ModuleRequest

**Required fix:** Define proper accessors for name, email, description. Add scopes. Add relationships.

---

#### MAJOR-5: Repair Migration Signals Broken Deployment

**File:** `database/migrations/2026_03_18_080000_repair_central_runtime_tables.php`

This migration recreates `cache`, `cache_locks`, `jobs`, `job_batches`, `failed_jobs`, and `sessions` tables with `Schema::hasTable` guards. The `down()` method is intentionally blank. Commit message: "fix(prod): repair central runtime tables and wait for mysql health."

This exists because production schema drifted away from migration history — tables were dropped manually, migrations were run out of order, or the migration table itself was corrupted. A repair migration is a band-aid over a broken deployment process.

**Required fix:** Document why this happened. Add a CI check to validate migration state. Mutations should run exactly once.

### Moderate Issues

#### MODERATE-1: hasPermission N+1 Trap

**File:** `app/Models/User.php:68`

`$this->loadMissing('role.permissions.feature')` is called inside `hasPermission()`, which runs in middleware on every request. `loadMissing` prevents duplicate loads, but the first call per request still triggers 2 extra queries. Any list view iterating users without eager-loading `role.permissions.feature` gets N+1 queries.

**Fix:** Eager load `role.permissions.feature` in user list queries. Cache the permission set per request lifecycle.

---

#### MODERATE-2: SyncPendingCloudflareDomain Chain Breaks Silently

**File:** `app/Jobs/SyncPendingCloudflareDomain.php:59`

The job chains itself by dispatching a new instance with incremented `pollAttempt`. If the queue worker crashes between the Cloudflare API call and the re-dispatch, the chain breaks permanently. The domain stays in a pending state forever with no recovery mechanism. The job has `$tries = 3`, but each "attempt" in the chain is a separate job — the retry count resets.

**Fix:** Add a scheduled command that finds domains stuck in pending state for longer than N minutes and re-dispatches sync jobs.

---

#### MODERATE-3: ModuleController Catches Throwable and Shows Error to User

**File:** `app/Http/Controllers/ModuleController.php:70`

```php
} catch (\Throwable $e) {
    return back()->withInput()->with('error', $e->getMessage());
}
```

Any exception, including internal server errors and sensitive filesystem exceptions, gets its raw message displayed to the user. A malicious ZIP that triggers a PHP error could leak server paths or other internal information.

**Fix:** Catch specific exception types. Log the full exception. Show a generic message to the user.

---

#### MODERATE-4: DomainCheckController Uses env() Directly

**File:** `app/Http/Controllers/DomainCheckController.php:25`

```php
$configuredToken = (string) env('DOMAIN_CHECK_TOKEN', '');
```

Using `env()` in application code instead of `config()` breaks when `php artisan config:cache` runs in production. After caching, `env()` returns null for anything not in the actual server environment variables.

**Fix:** Add `DOMAIN_CHECK_TOKEN` to a config file and use `config('domain_check.token')`.

---

#### MODERATE-5: DomainController is a God Controller

**File:** `app/Http/Controllers/Tenant/DomainController.php` — 322 lines

The controller handles listing, creating, showing, storing, status checking, DNS verification, and deleting domains all in one file. The `store` method alone is 68 lines with inline validation, Cloudflare integration, and error handling. This is a service masquerading as a controller.

The `verify` method has three different code paths depending on whether Cloudflare is involved, whether the domain has a hostname ID, and whether DNS verification passes. Testing any individual path requires mocking the other paths.

**Fix:** Extract domain creation into an action class. Extract verification into a dedicated service method. Keep the controller to routing glue.

---

#### MODERATE-6: Action Classes Used Inconsistently

Tenant creation and update use Action classes (`CreateTenantAction`, `UpdateTenantAction`), but module install/uninstall logic lives in the controller dispatching jobs directly. Domain creation has no action class. The module system has zero action classes.

**Fix:** Either commit to the Action pattern across all resources or drop it. Half-adoption is worse than no pattern at all.

## Docker & Infrastructure Audit

### High-Severity Issues

#### INFRA-HIGH-1: Production Compose Bind-Mounts Entire Source

**File:** `docker-compose.prod.yml:14-15`

```yaml
volumes:
    - ./:/var/www
```

The production compose mounts the entire project directory (including `.env` with secrets, `.git`, and `storage/*.key`) into the container. This defeats the multi-stage Dockerfile build, which produces a clean production image in the `production` stage. The bind-mount means the Dockerfile's `composer install --no-dev`, `npm run build`, and `rm -rf node_modules .git .github tests` are all ignored — the host directory overwrites the baked-in image.

**Fix:** Remove the bind-mount. Use the production Docker image's built-in `/var/www`. If hot-reload of config/secrets is needed, mount only specific files, not the entire project root.

---

#### INFRA-HIGH-2: Container Runs as Root

**File:** `DockerFile:43`

```dockerfile
CMD ["php-fpm"]
```

The Dockerfile never creates a non-root user and never issues a `USER` directive. PHP-FPM runs as `root` inside the container. The entrypoint script (`docker/prod/entrypoint.sh`) has logic to drop privileges via `gosu` to `www-data`, but it is never wired into the image — there is no `ENTRYPOINT` directive.

If the container is compromised (e.g., via the ZIP RCE), the attacker gets root access inside the container.

**Fix:** Add `RUN apt-get install -y gosu` to the Dockerfile. Add `ENTRYPOINT ["docker/prod/entrypoint.sh"]` before `CMD`.

---

#### INFRA-HIGH-3: No .dockerignore

The project has no `.dockerignore` file. `COPY . .` in the Dockerfile builder stage copies everything into the build context, including:

- `.env` with live Cloudflare API tokens and database passwords
- `.git/` directory with full commit history
- `node_modules/` and `vendor/` (bloating the context, though overwritten by composer/npm)
- `storage/logs/`, `storage/framework/`

Secrets copied into a Docker layer are permanently stored in the image history and can be extracted by anyone with access to the image.

**Fix:** Create a `.dockerignore` at the project root excluding `.env`, `.git/`, `node_modules/`, `vendor/`, `storage/`, `tests/`, `*.md`, `.github/`, etc.

---

#### INFRA-HIGH-4: No Security Headers in Nginx

**Files:** `docker/nginx/conf.d/app.conf` (dev) and `docker/nginx/conf.d/prod/app.conf` (prod)

Neither nginx configuration sets any security headers. Missing at minimum:

- `Strict-Transport-Security` (prevents downgrade attacks)
- `X-Frame-Options` (prevents clickjacking)
- `X-Content-Type-Options` (prevents MIME sniffing)
- `X-XSS-Protection` (XSS auditor)
- `Content-Security-Policy` (mitigates XSS and data injection)
- `Referrer-Policy` (controls referrer information leakage)

Production HTTPS server (port 443) is the most critical: without `Strict-Transport-Security`, browsers won't enforce HTTPS, leaving users vulnerable to SSL stripping.

**Fix:** Add a security headers include file and reference it from both server blocks.

---

#### INFRA-HIGH-5: No Scheduler Service

Neither `docker-compose.yml` nor `docker-compose.prod.yml` defines a scheduler container. Laravel 12 requires either `php artisan schedule:work` running as a persistent process, or a system crontab with `* * * * * php artisan schedule:run`.

Currently there are no scheduled tasks, so this does not cause immediate breakage. But when scheduled tasks are added (e.g., periodic Cloudflare domain sync sweep, session cleanup), they will silently not run.

**Fix:** Add a lightweight scheduler service to both compose files running `php artisan schedule:work`.

---

#### INFRA-HIGH-6: No Static Asset Caching or Gzip in Nginx

Neither nginx config has:

- A `location ~* \.(css|js|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$` block with `expires` and `Cache-Control: public, max-age=31536000, immutable`
- `gzip on;` with `gzip_types` for text assets

Vite-generated assets have content hashes in their filenames (e.g., `app-8GCeNMCe.js`), making them perfect candidates for immutable long-term caching. Every page load currently re-serves these files without caching headers, wasting bandwidth and slowing page loads.

**Fix:** Add a static asset location block with aggressive caching. Enable gzip compression for text-based MIME types.

---

### Medium-Severity Issues

#### INFRA-MED-1: No OPcache Extension

**File:** `DockerFile:18-24`

The Dockerfile installs `pdo_mysql`, `bcmath`, `intl`, `zip`, and `gd` but not `opcache`. OPcache stores precompiled PHP bytecode in shared memory, providing 30-50% throughput improvement for production PHP-FPM workloads.

**Fix:** Add `docker-php-ext-install opcache` and configure `opcache.ini`.

---

#### INFRA-MED-2: No Resource Limits

Neither compose file sets `deploy.resources.limits` or `mem_limit` on any service. A runaway queue worker or a query that exhausts MySQL buffer pool memory can consume all host resources. MySQL in particular should have at minimum a memory limit to prevent OOM kills.

**Fix:** Add `deploy.resources.limits.memory` to `mysql`, `queue`, and `app` services.

---

#### INFRA-MED-3: No Queue Worker Health Check

The `queue` service has no `healthcheck` directive. If the queue worker silently crashes (e.g., after a fatal PHP error in a job), Docker won't detect it unless the process itself exits.

**Fix:** Add a health check that verifies `php artisan queue:monitor` or checks that the worker process is still alive.

---

#### INFRA-MED-4: No Docker Build in CI

**File:** `.github/workflows/ci.yml`

CI runs tests and builds frontend assets but never validates the Docker image. A broken Dockerfile would only be detected at deploy time.

**Fix:** Add a `docker build -t app-test .` step to CI.

---

#### INFRA-MED-5: No Pint or Security Scanning in CI

CI does not run `./vendor/bin/pint --test` or `composer audit`. Code style violations and known-composer-vulnerabilities are only caught manually.

**Fix:** Add pint and composer audit steps to CI.

## Deployment Readiness Audit

### FAIL: 22 of 52 Tests Failing

**Command:** `php artisan test --compact`
**Result:** 30 passed, 22 failed (89 assertions)

Failure categories:

1. **SQLite transaction conflicts:** Multiple tests fail with `PDOException: There is already an active transaction`. Affects `CloudflareDomainStatusSyncTest`, `TenantDomainLifecycleTest`, `TenantOnboardingTest`. Root cause: SQLite in-memory databases with multiple connections or nested transactions.

2. **Test-reality mismatch:** `ExampleTest` expects redirect to `/tenants` but actual redirect goes to `/dashboard`. The auth redirect logic changed but the test was not updated.

3. **Stale mocks:** `CloudflareDomainStatusSyncTest` mocks `CloudflareService` but the controller constructor expects `DomainCloudflareSyncService`. The test was not updated after a service refactor.

### WARN: route:cache Fails

```
LogicException: Unable to prepare route [dashboard] for serialization.
Another route has already been assigned name [dashboard].
```

**Root cause:** `routes/web.php:15` defines `->name('dashboard')` and `routes/tenant.php:40` also defines `->name('dashboard')`. When routes are serialized for caching, duplicate names cause a fatal error. `php artisan route:cache` cannot be used in production until this is fixed, which has a performance impact.

**Fix:** Rename the tenant dashboard route to `tenant.dashboard`.

### WARN: Missing Environment Variables

`.env.example` is missing ~10 environment variables that are referenced in config files:

| Missing Variable                 | Used In                                   |
| -------------------------------- | ----------------------------------------- |
| `CLOUDFLARE_ZONE_DOMAIN`         | `config/cloudflare.php:13`                |
| `SESSION_SECURE_COOKIE`          | `config/session.php:172`                  |
| `SLACK_BOT_USER_OAUTH_TOKEN`     | `config/services.php:33`                  |
| `SLACK_BOT_USER_DEFAULT_CHANNEL` | `config/services.php:34`                  |
| `LOG_SLACK_USERNAME`             | `config/logging.php` (slack channel)      |
| `LOG_SLACK_EMOJI`                | `config/logging.php` (slack channel)      |
| `LOG_STDERR_FORMATTER`           | `config/logging.php` (stderr channel)     |
| `LOG_SYSLOG_FACILITY`            | `config/logging.php` (syslog channel)     |
| `PAPERTRAIL_URL`                 | `config/logging.php` (papertrail channel) |
| `PAPERTRAIL_PORT`                | `config/logging.php` (papertrail channel) |

### WARN: Weak Default Passwords

- `config/auth.php:24`: `CENTRAL_SUPERADMIN_PASSWORD` falls back to `'password'`
- `TenantBootstrapSeeder.php:19`: `TENANT_DEFAULT_ADMIN_PASSWORD` defaults to `'ChangeMe123!'`

Both are present in `.env.example` with these weak values. If an operator copies `.env.example` to `.env` without changing passwords, production runs with trivially guessable credentials.

### WARN: SESSION_SECURE_COOKIE Not Defaulted

`config/session.php:172` reads `env('SESSION_SECURE_COOKIE', null)`. On HTTPS production sites, this must be `true` to prevent session cookie transmission over HTTP (session hijacking). Setting it to `null` leaves the decision to Laravel's heuristics.

### WARN: No Scheduled Tasks

`routes/console.php` defines two Artisan commands (`inspire`, `domains:sync-cloudflare`) but neither is scheduled. No periodic Cloudflare domain sweep exists. Production still needs the standard `* * * * * php artisan schedule:run` cron configured, even if empty.

## What Passed

Despite the issues above, many things are done correctly:

- **Tenancy foundation:** Stancl Tenancy with domain identification, separate tenant databases, proper bootstrappers. Config is well-structured.
- **Job resilience:** All queue jobs implement `ShouldQueue` with explicit `$tries`, `$timeout`, `backoff()`, and `failed()` handlers. Tenant-aware jobs properly initialize/tear down tenancy context in try/finally.
- **Logging:** `AddTenantContext` processor adds `tenant_id`, `host`, `context`, `request_id`, and `job_id` to every log record — excellent for production debugging.
- **Telescope:** Properly gated by email allowlist, disabled by default, redacts sensitive request data in non-local environments.
- **Tests exist:** 52 tests covering tenant provisioning E2E flow, module lifecycle, Cloudflare status sync (with mocks), domain checking, and host access policy. They just need fixing.
- **Multi-stage Dockerfile:** The three-stage build (base → builder → production) is correct. It just needs the entrypoint wired up and the bind-mount removed.
- **Error handling:** `APP_DEBUG` defaults to `false`, `APP_ENV` defaults to `production`, `TenantCouldNotBeIdentifiedOnDomainException` returns generic 404.
- **Migrations:** Well-structured. Tenant migrations create the same cache/jobs tables as central for tenant DB isolation. The repair migration is safely idempotent with `Schema::hasTable()` guards.
- **CI exists:** GitHub Actions runs `composer validate`, `npm ci`, `npm run build`, and `php artisan test` on every push. Needs a Docker build step and pint/composer audit.
- **Tenant host validation:** `RejectInvalidTenantHost` middleware properly rejects central hosts (404), unknown hosts (404), and unverified hosts (403).

4. Move module state from JSON blob on tenant record to dedicated pivot table
